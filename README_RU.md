# Go API Gateway (RU)

HTTP API-шлюз с конфигурационным API, reverse proxy и логированием запросов в PostgreSQL.

> Это учебный проект, используемый для тестирования и отладки воркфлоу работы с AI‑ассистентами.

## Предварительные требования

- Go 1.22+
- PostgreSQL (для хранения: маршрутов, глобальной конфигурации заголовков, логов запросов)

## Переменные окружения

Шлюз читает конфигурацию из переменных окружения. Они также автоматически подгружаются из файла `.env` в корне проекта (через `godotenv`), если он существует.

| Переменная | Обязательна | Значение по умолчанию | Описание |
|-----------|-------------|-----------------------|----------|
| `PORT` | Нет | `8080` | Порт HTTP‑сервера. |
| `DATABASE_URL` | Да | — | Строка подключения к PostgreSQL, например `postgres://user:pass@localhost:5432/gateway?sslmode=disable`. |
| `CORS_ALLOWED_ORIGIN` | Нет | — | Один origin, которому разрешён CORS (например, `https://app.example.com`). Если пусто — CORS‑заголовки не ставятся. **Никогда не используйте `*` в продакшене.** |

Пример `.env` (см. также `.env.example`):

```env
PORT=8080
DATABASE_URL=postgres://gateway:gateway@localhost:5432/gateway?sslmode=disable
CORS_ALLOWED_ORIGIN=https://app.example.com
```

## Миграции БД

Перед запуском шлюза примените миграции (из корня проекта):

```bash
go run ./cmd/migrate up
```

Для отката:

```bash
go run ./cmd/migrate down
```

Порядок миграций: `000001` (routes), `000002` (request_logs), `000003` (global_header_config). Подробности — в `docs/create/go-api-gateway/phase/migrations.md`.

## Запуск шлюза

Из корня проекта:

1. Настройте `.env` или экспортируйте переменные окружения.
2. Запустите:

```bash
go run ./cmd/gateway
```

Пример с инлайновыми переменными:

```bash
PORT=8080 DATABASE_URL="postgres://user:pass@localhost:5432/gateway?sslmode=disable" go run ./cmd/gateway
```

Пример сборки бинарника:

```bash
go build -o gateway ./cmd/gateway
export PORT=8080
export DATABASE_URL="postgres://user:pass@localhost:5432/gateway?sslmode=disable"
./gateway
```

Сервер слушает порт `PORT`, корректно завершает работу по SIGINT/SIGTERM и обрабатывает:

- `GET /` — health‑чек (`gateway ok`).
- **Config API** под `/api`.
- **Proxy** для всех остальных путей (по настроенным маршрутам).

## Обзор API

### Config API (базовый путь `/api`)

- **GET /api/routes** — список маршрутов.
- **POST /api/routes** — создание маршрута (тело: `name`, `path` или `path_prefix`, `upstream`, опционально `headers_to_forward`, `headers_to_set`).
- **GET /api/routes/:id** — получение маршрута по ID.
- **PUT /api/routes/:id** — полное обновление маршрута.
- **DELETE /api/routes/:id** — удаление маршрута.
- **GET /api/config/headers** — получение глобальной конфигурации заголовков.
- **PUT /api/config/headers** — установка глобальной конфигурации заголовков.

Тела запросов/ответов — JSON. Полный контракт (валидация, коды ответов, конфликты) описан в `docs/create/go-api-gateway/design/api.md`.

### Поведение прокси

- Все запросы, **не** начинающиеся с `/api`, сопоставляются с маршрутами (побеждает самый длинный путь/префикс).
- Соответствующие запросы проксируются на `upstream` (метод, полный путь, query, тело) с таймаутом 30 секунд.
- Заголовки:
  - Из входящего запроса пересылаются только те, чьи имена есть в объединённом списке `headers_to_forward` (глобальные + маршрута).
  - Заголовки из `headers_to_set` применяются поверх (сначала глобальные, потом маршрута, маршрут перекрывает глобальные).
- Каждый проксируемый запрос логируется в таблицу `request_logs` (метод, путь, имя маршрута, статус, длительность, заголовки). Чувствительные значения заголовков перед сохранением редактируются (redact).

Подробности — в `docs/create/go-api-gateway/design/proxy.md`.

## Rate limiting

- **Глобально:** 100 запросов в час на один IP (proxy и `/`).
- **Config API (`/api/*`):** 20 запросов в час на один IP.

При превышении лимита шлюз возвращает **429 Too Many Requests**. IP берётся из `X-Forwarded-For` (первый адрес), иначе `RemoteAddr`.

## Замечания по безопасности

- Секретов в коде нет; используйте переменные окружения (`DATABASE_URL`, `CORS_ALLOWED_ORIGIN` и т.д.).
- CORS: только настроенный origin, `*` не используется.
- Все обращения к базе используют параметризованные запросы; в логах запросов хранятся только «очищенные» заголовки (без реальных значений `Authorization`, `Cookie`, API‑ключей и т.п.).
- Аутентификация для Config API в этом варианте не реализована — для продакшена нужно ограничить сетевой доступ или добавить auth (например, API‑ключи или внешнего провайдера).

## Тесты

```bash
go test ./...
```

Тесты для config/migrate могут пропускаться, если `DATABASE_URL` не задан. Интеграционные тесты для API и proxy требуют запущенный Postgres и корректный `DATABASE_URL`, указывающий на тестовую БД.
